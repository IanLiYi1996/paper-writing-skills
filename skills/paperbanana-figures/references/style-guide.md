# NeurIPS 2025 Figure Style Guide: Soft Tech & Scientific Pastels

This guide defines the visual language for publication-quality figures targeting NeurIPS, ICML, ICLR, and related ML venues. All figures generated by this skill must comply with these standards.

---

## Color Palette

### Background Colors (10-15% Opacity Pastels)

Use these as fill colors for grouping boxes and background regions. They should be barely visible -- just enough to delineate boundaries.

| Purpose              | Hex Code  | Description              |
|----------------------|-----------|--------------------------|
| Primary group        | #E8F4FD   | Very light blue          |
| Secondary group      | #FFF3E0   | Very light orange        |
| Tertiary group       | #E8F5E9   | Very light green         |
| Quaternary group     | #F3E5F5   | Very light purple        |
| Neutral group        | #F5F5F5   | Very light gray          |
| Highlight region     | #FFFDE7   | Very light yellow        |

### Active Element Colors (Medium Saturation)

Use these for boxes, nodes, and other foreground elements. They carry semantic meaning.

| Purpose              | Hex Code  | Description              |
|----------------------|-----------|--------------------------|
| Trainable component  | #4A90D9   | Medium blue (warm-cool)  |
| Frozen component     | #7FB3D8   | Light steel blue         |
| Loss / objective     | #E07A5F   | Terracotta orange        |
| Data / input         | #81C784   | Medium green             |
| Output / result      | #9575CD   | Medium purple            |
| Attention / key      | #FFB74D   | Medium orange            |
| External system      | #90A4AE   | Blue gray                |
| Novel contribution   | #FF8A65   | Bright coral (emphasis)  |

### Complementary Pairings

Use these pairings when you need two contrasting colors side by side:

- **Blue / Orange**: `#4A90D9` / `#E07A5F` -- Best for trainable vs. frozen, or method vs. baseline.
- **Green / Purple**: `#81C784` / `#9575CD` -- Best for input vs. output, or encoder vs. decoder.
- **Teal / Coral**: `#26A69A` / `#FF8A65` -- Best for novel vs. existing components.

## Typography

### Labels and Titles

- **Font family**: Sans-serif (Arial, Roboto, Helvetica, or DejaVu Sans).
- **Module labels**: 10-12pt, regular weight, `#333333`.
- **Group titles**: 10-12pt, semibold, `#333333`.
- **Arrow labels**: 8-10pt, regular weight, `#666666`.
- **Annotations**: 8-9pt, italic, `#666666`.

### Mathematical Notation

- **Font family**: Serif (Times New Roman, Computer Modern, or STIX).
- **Style**: Always italic for variables: *x*, *W*, *h_t*.
- **Placement**: Inside or adjacent to the relevant element, never floating.

### Rules

- Never mix more than 2 font families in a single figure.
- All text at the same hierarchical level must use the same font size.
- Minimum font size: 7pt at final print size. If a label cannot fit at 7pt, the element is too small.

## Line and Arrow Styles

### Connection Types

| Context                     | Line Style   | Arrowhead     | Color     |
|-----------------------------|-------------|---------------|-----------|
| Data flow (forward pass)    | Solid       | Filled triangle | #555555 |
| Gradient flow (backward)    | Dashed      | Open triangle   | #AAAAAA |
| Optional / conditional      | Dotted      | Filled triangle | #888888 |
| Residual / skip connection  | Solid       | Filled triangle | #4A90D9 |
| Bidirectional               | Solid       | Both ends       | #555555 |

### Routing

- **Architecture diagrams**: Use orthogonal (elbow) paths. All turns are 90 degrees. Arrows run along grid lines.
- **System / logic diagrams**: Curved (bezier) paths are acceptable for showing logical flow.
- **Never**: Diagonal arrows that cut across elements at arbitrary angles.

### Arrow Spacing

- Arrows should not touch their source/target elements. Leave 3-5px of clearance.
- Parallel arrows must be evenly spaced.
- Crossing arrows should be minimized. If unavoidable, add a small gap or bridge at the crossing point.

---

## Layout Principles

### Grid Alignment

- All elements must snap to a virtual grid. A 20px grid is recommended.
- Horizontal and vertical centers of same-level elements must be aligned.
- Consistent spacing between elements: use the same gap (e.g., 40px) throughout.

### Flow Direction

- **Default**: Left-to-right for pipelines and sequential processes.
- **Alternative**: Top-to-bottom for hierarchical architectures.
- **Never**: Mix flow directions in the same diagram. If a feedback loop exists, route it along the edges (not through the middle).

### Whitespace

- At least 20% of the canvas should be whitespace.
- Group boundaries should have internal padding of at least 15px.
- Do not fill every available space -- breathing room aids readability.

### Visual Hierarchy

- The most important element (novel contribution) should be: largest, most saturated, or centrally placed.
- Supporting elements should be smaller or less saturated.
- Background grouping boxes should recede visually (very light fill, thin or no border).

## Common Pitfalls to Avoid

1. **PowerPoint defaults**: Drop shadows, 3D bevels, gradient fills, and WordArt have no place in academic figures.
2. **Saturated colors**: Pure `#FF0000`, `#00FF00`, `#0000FF` look unprofessional. Always desaturate.
3. **Font mixing**: Using 3+ font families creates visual chaos. Stick to one sans-serif plus one serif.
4. **Inconsistent arrows**: Mixing curved and orthogonal arrows in the same diagram.
5. **Decorative elements**: Clip art, icons from the web, stock photo backgrounds.
6. **Over-labeling**: Putting too much text in the diagram. If an element needs a paragraph of explanation, it belongs in the caption.
7. **Color-only encoding**: Using only color to distinguish categories. Always add a secondary visual channel (shape, pattern, line style).
8. **Tiny text**: Any text that becomes unreadable when the figure is scaled to column width.
